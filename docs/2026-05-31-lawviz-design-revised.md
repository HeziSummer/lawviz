# LawViz 产品设计规格文档（修订版）

**版本**：v1.1 修订版  
**日期**：2026-05-31  
**阶段**：灰测版（一期）  
**状态**：设计定版待实现

> 本文档基于 v1.0 的审计和深度讨论，修正了原设计中的矛盾和不合理之处。

---

## 执行摘要

**产品定位**：律析 LawViz — 律师用 AI 生成案件可视化 HTML 报告的 Web SaaS，灰测版给朋友玩，正式版走积分制付费。

**核心改变**（vs v1.0）：
1. **生成方式**：改为预设模板 + 数据填充（而非 LLM 自由生成 HTML）
2. **结果页架构**：HTML 通过自己的域名代理（而非直连 OSS iframe），解决编辑+截图问题
3. **层级分法**：三个独立的付费服务（GPT / Claude / 法宝 MCP），各自独立计费和折扣
4. **模板设计**：每个案件类型需要与产品负责人逐一深度讨论，确定该类最值得可视化的内容
5. **费用框架**：不写死具体积分数字，改为灰测期间通过实际成本验证后反推定价

## 3. 产品分层 & 定价模型

### 3.1 三个独立的付费服务

**用户可任意组合，各自独立计费、独立折扣**：

1. **GPT-5.5 生成**：快速、成本最低
2. **Claude Opus 生成**：更深度分析、成本中等
3. **北大法宝 MCP 查询**：法律准确性、成本最高

### 3.2 三档用户权益

| 权益项 | tier1 (免费/尝鲜) | tier2 (标准) | tier3 (专业) |
|--------|---|---|---|
| **初始积分** | 10 | 10 | 10 |
| **GPT 生成** | ✓ 基础价 | ✓ 基础价 | ✓ 打 7.5 折 |
| **Claude 生成** | ✗ | ✓ 可选，扣更多积分 | ✓ 打 8 折 |
| **法宝查询** | ✗ | ✓ 可选，扣更多积分 | ✓ 打 8 折 |
| **组合方式** | GPT only | 可分别选 Claude 或法宝 | 各自独立打折，无折扣叠加 |

**关键约束**：tier3 用户既能用便宜的 GPT（打折），也能用贵的 Claude + 法宝（各自打折），符合"贵的包含便宜的"原则。

## 4. 对话成本可控原则

**核心约束**：既然采用积分制，必须确保用户在当前 tier 的积分范围内完成对话，不被突然超支卡住。

### 4.1 定价基准

- **GPT 基础生成 = X 积分**（参考价，具体数字灰测后确定）
- 这个价格对应"标准案情、标准追问"的平均成本

### 4.2 动态成本调整

**实际费用 = 基础价 × 模型系数 + 复杂度增加费**

**复杂度因子**：
- **输入超长**：案情/关键事实 > 2000 字 → 费用增加（幅度待定）
- **追问超过基础轮数**：第 1-2 轮免费；第 3-5 轮各扣 0.1 积分（具体价格灰测后确定）
- **多服务组合**：Claude + 法宝分别计费，无折扣叠加

### 4.3 用户体验保障

1. **生成前预估**：系统显示 "预计消耗 X.XX 积分"（基于当前输入 + 预期追问）
2. **追问前告知**：每轮追问前弹窗 "这一轮会扣 Y 积分，确认继续？"
3. **失败退款**：生成失败，立即全额退款
4. **输入截断**：超过长度限制时，前端提示并预估费用增加幅度，用户可选择缩短或支付

## 5. 生成流程 & 模板设计框架

### 5.1 生成架构

```
结构化表单填写（必填项）
  ↓
AI 智能追问（识别遗漏的关键信息，<=5 轮）
  ↓
聚合：输入数据 + 追问历史 + tier 信息 + 成本因子
  ↓
LLM 生成该案件类型的"动态分析内容"
  （内容形式和重点由该案件类型的模板规范决定）
  ↓
数据 + 动态内容 → 套进"该案件类型的预设 HTML 模板"
  ↓
最终 HTML（样式和结构固定，数据和分析可变）
```

### 5.2 模板设计 = 产品设计的核心

**每个案件类型都需要独立设计**，包含：

1. **案件分析**（产品负责人 + Agent）：
   - 该案件类型的律师工作流是什么？
   - 最关键的决策点是什么？
   - 哪些信息最值得在 HTML 里可视化？

2. **可视化设计**（产品负责人 + Agent）：
   - 该类案件的 HTML 骨架应该强调什么结构？
   - 哪些区块是必须的、哪些是可选的？
   - 数据和分析内容怎么映射进去？

3. **交付物**：
   - HTML 骨架（Jinja2 模板，含占位符）
   - 字段映射表（表单数据 → HTML 节点）
   - LLM 指导文档（要生成什么内容、分析重点、风格要求）

**5 个初始案件类型**：合同纠纷、劳动争议、婚姻家事、交通事故、刑事辩护

## 6. 费用模型框架 & 灰测成本验证

### 6.1 费用不写死，走灰测反推

**原因**：
- 不知道 "标准合同纠纷生成一次需要多少 token"
- 不知道 "法宝调用一次的实际成本"
- 不知道 "Claude vs GPT 的成本倍数"

**方法**：灰测期间记录每次生成的实际成本，灰测后期反推定价。

### 6.2 灰测数据收集清单

每次生成记录：
- 案件类型 + 细分案由
- 输入字数 + 追问轮数
- 使用的模型（GPT / Claude）
- 是否调用了法宝，调用了几次，返回了几条结果
- LLM token 消耗（输入 + 输出）
- 生成耗时
- 实际成本（如果有 API 计费单据）

### 6.3 冻结费用表的时间点

**灰测结束时（预计 10 月初）**：
- 分析成本分布（标准 vs 复杂案情）
- 计算平均成本
- 反推定价（基础价 × 3 倍利润率）
- 冻结费用表
- 正式版上线前最后确认一次

## 7. 结果页架构 & 编辑+截图实现

### 7.1 问题：跨域 iframe 的限制

原设计直接用 `<iframe src="OSS域名">`，导致：
- 父页面无法读写 iframe 内容 → 编辑功能做不了
- html2canvas 无法截取跨域 iframe → 截图功能做不了

### 7.2 解决方案：自己的域名代理

```
律师浏览器 
  → yourdomain.com/api/render/{gen_id}
  → 服务器从 OSS 拿 HTML
  → 服务器返回给浏览器
  → 浏览器看到的 HTML 来自 yourdomain.com（同源！）
  → 编辑+截图都能做
```

**成本影响**：基本零（网络往返多一次，但内网转发，几乎无额外成本）

### 7.3 结果页功能

**左侧预览（70%）**：
- `<div id="preview">` 直接渲染 HTML（不用 iframe）
- 支持 inline 编辑：点击 `.editable` 元素 → contenteditable=true → 失焦时 PATCH `/api/generate/{gen_id}/html`

**右侧操作面板（30%）**：
- 积分消耗说明
- 使用模型标注（GPT / Claude / 法宝）
- "下载 HTML"
- "复制分享链接"
- "截图下载"（html2canvas，同源，可用）
- "重新生成"（扣积分重新走流程）
- tier3 用户：展开法宝引用清单

## 8. 前端页面框架

**原则**：桌面版（1280px），响应式留给二期小程序

### 8.1 落地页（`/`）

未登录显示。已登录重定向 `/dashboard`。

**内容**：
- Hero：产品价值主张 + "立即体验" CTA
- 三步流程展示（填写 → 生成 → 分享）
- 5 个模板预览卡片
- 积分包价格展示（尝鲜 / 标准 / 专业）
- 底部联系方式

### 8.2 仪表盘（`/dashboard`，需登录）

- 积分余额 + "充值"按钮
- "新建可视化"大按钮
- 最近 10 次生成历史（分页）：模板名、tier、生成时间、状态、"查看"/"分享"

### 8.3 模板选择（`/generate`，需登录）

- 5 个模板卡片网格
- 点击卡片 → 弹出 tier 选择 Modal
- 每个 tier 展示价格/权益/建议
- 确认 tier → 跳转 `/generate/{templateId}?tier={tier}`

### 8.4 生成流程（`/generate/[templateId]`，需登录）

同一页面内三个阶段切换（state 控制）：

**阶段 1：填表**
- 根据模板的 fields_schema 动态渲染表单
- 必填字段红色标注
- "下一步"验证必填项 → 调用 `POST /api/generate/start`

**阶段 2：AI 追问**
- QAChat 组件：展示 AI 问题和用户回答历史
- 输入框 + "提交回答"按钮
- 显示"第 X 轮，最多 5 轮"
- "跳过，直接生成"按钮（可随时点击）
- AI 返回 READY 自动进入阶段 3

**阶段 3：生成中**
- 进度提示 + 加载动画
- 每 2 秒轮询 `GET /api/generate/{gen_id}/status`
- 完成 → 自动跳转 `/generate/result/{genId}`

### 8.5 结果页（`/generate/result/[genId]`，需登录）

见第 7 节"结果页架构"。

### 8.6 分享页（`/share/[token]`，无需登录）

- 全屏展示 HTML
- 顶部小横幅："由 LawViz 生成 · 立即试用"

## 10. API 设计概览

### 10.1 认证

```
POST /api/auth/register  {phone, password} → {user_id, token, credits}
POST /api/auth/login     {phone, password} → {user_id, token, credits}
GET  /api/auth/me        [需认证] → {user_id, phone, credits, ...}
```

### 10.2 模板

```
GET /api/templates
GET /api/templates/{template_id}
```

### 10.3 生成流程

```
POST /api/generate/start        {template_id, tier, input_data} → {gen_id}
POST /api/generate/{gen_id}/qa  {answer} → {status, question?, ready?}
POST /api/generate/{gen_id}/submit → {gen_id, status}
GET  /api/generate/{gen_id}/status → {status, html_url?, share_token?, ...}
PATCH /api/generate/{gen_id}/html {html} → {success}
GET  /api/generate/history      [分页] → {total, items}
```

### 10.4 积分

```
GET /api/credits/balance
GET /api/credits/packs
POST /api/credits/purchase {pack_id} → {pay_url, order_id}
POST /api/webhooks/hupijiao [签名验证] → 积分到账
```

### 10.5 HTML 代理

```
GET /api/render/{gen_id}  → 返回 HTML（用于 iframe/直接 div 渲染）
```

### 10.6 公开分享

```
GET /api/share/{share_token}  → {template_name, tier, html_url, created_at}
```
